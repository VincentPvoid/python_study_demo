## fp文件分割
文件夹在 Res\x64\sound\voice 下，从文件夹名称很容易能看出是声音文件  
都是fp格式  

老方法扔进 十六进制编辑器 里查看  
能看出来里面都是一段一段的文件，中间用一堆 `00` 隔开，应该是拼起来的音频  
开头都是 `KOVS`，即 `4B 4F 56 53`（在文件中间不会出现）  
结尾不固定，`00`的数量也不固定  

所以可以根据`KOVS`的位置来确定分割点  
遍历到当前的`KOVS`位置后，因为中间有`00`，因此上一个文件尾的位置可以往前取一点  
一开始是往前取了32位，但检查分割后的文件文本结尾，发现和`KOVS`数量对不上，检查了一下发现中间有一条分隔的`00`位数不足，因此改成了往前取8位，这样就肯定不会截断到非`00`的位置  


或者可以根据文件大小来取  
`KOVS`后面2位是文件大小，根据小端序转换，就是当前文件块大小  

总之两种方法都可以，都可以提取，应该提取出来的文件也是一样的  
因为文件太多了没全部听完，随便点开都可以听，文件数量也没问题，所以应该两种提取方法都可以


### 踩坑
看了下大部分文件块的结束位置都是 `0E00000000`，所以一开始以这个作为结束标识，但检查分割后文件数量发现与`KOVS`数量不符  
再次检查原.fp文件，发现有些并不符合该结尾，需要用其他更匹配的方法进行分割  




## 解密文件
观察分割后的文件头，跳过包括`KOVS`在内的32位，从第33位开始都是统一的 `OfeP`，即`4F 66 65 50 04`  
音频文件中也多次出现了 `OggS`（即 `4F 67 67 53`）
通过各种搜索查看参考和询问AI得知，该公司游戏（特别是比较旧的游戏）会使用简单的异或加密，而且经大概率只会加密前 256 或 1024 位
以`OfeP`为例
```
第 0 字节：4F ^ 0 = 4F (对应字符 O)
第 1 字节：66 ^ 1 = 67 (对应字符 g)
第 2 字节：65 ^ 2 = 67 (对应字符 g)
第 3 字节：50 ^ 3 = 53 (对应字符 S)
```
总之就是个简单异或，就是猜到底加密了前多少位  
这次的很幸运，只试了256位就提取出来了
